    Lab 4 - OpenMP

    Due: Wednesday, February 13, 10 p.m.

    Points: 1.5

    Deadline to find your group partner and create your group on MarkUs: Wednesday, February 13, at 2:10pm. Your group partners for assignment 3 and lab 4 remain the same.

    Scinet

    We will use Scinet for this lab. You should have already received an email with your Scinet login. In that email you see a link to The short intro to Scinet document (the document is also uploaded in Quercus). You have to read that document before starting this lab. While you can use your own machine or a lab machine to debug the code, your final code has to be performance engineered and executed on Scinet. We will do all grading on Scinet. For each part, we do provide scripts to help with running your code on Scinet.

    1. Introduction

    This exercise is intended to help you get some practice with OpenMP directives. Please start early to fully take advantage of the lab time to ask questions. We highly encourage that you read the OpenMP tutorial (Links to an external site.)Links to an external site. from the Lawrence Livermore National Laboratory before starting or at least use it as a reference.

    You should work in pairs for this exercise and only with your Assignment 3 partner, since some code design may potentially be common with the assignment. Please log into MarkUs well before the deadline to take these steps, find your repository and make sure that you can commit and push to it. Do not create a separate directory in your repository, it should already be created for you. (If you create the directory manually, then MarkUs won't know about it and we won't be able to see your work.)

    It is your responsibility to log into MarkUs before the exercise deadline to ensure that you know where to commit/push your work, and so that MarkUs can connect your work to the grading system.

    2. Requirements

    You can find the starter code on Scinet under /home/t/teachcsc367/CSC367Starter/labs/lab4/starter_code.tgz  so copy this into your repository and make sure you can do your first commit. Please make sure to read carefully over the code, including the licenses and the instructions from the comments.

    Your task for this exercise is to work on implementing two short pieces of code in OpenMP. The goal is for you to get some hands-on experience with OpenMP directives and time your parallel code.

    Part1: Parallelizing computation using OpenMP

    In this part, your task is to parallelize the code that computes an integral of a function over a given interval, using OpenMP. The sequential program that computes the integral is provided in integral-seq.c in the starter code. Your task is to implement the parallel version using OpenMP in integral-omp.c.
    Hint: using a simple "parallel for" with the unmodified for loop in the starter code will not work (it won't compile; see the compiler error message to understand what the problem is). You will have to adjust the code so that it can be easily parallelized with a "parallel for" clause.
    You must also measure the execution time (in milliseconds), in both the sequential and the parallel versions (see comments in the starter code). You must keep the same output format as in the starter code; do not add any new output, otherwise the autotester will fail your submission.

    Part2: Scheduling parallel tasks and profiling

    In this part, you are to implement code that calculates how many prime numbers exist, which are smaller than a given N (passed as a command line argument). Your code will test all the numbers up to N for prime-ness, and count the total number of primes found. The search for primes can be done in parallel by assigning each thread a chunk of numbers to test. The sequential version of the program is implemented in primes-seq.c.
    NOTE: do not change the way the numbers are tested for prime-ness (the is_prime() function), otherwise the execution times will be off, and the autotester will fail your submission.

    You are to parallelize the program using OpenMP and test several ways to schedule the tasks, using either the default scheduling, or using an explicit clause:

    Use the default scheduling (implement it in primes-default.c) Statically, using a schedule(static) clause (implement it in primes-static.c) Dynamically, using a schedule(dynamic) clause (implement it in primes-dynamic.c). For the latter two you are required to vary the chunk size, to see what impact it has on performance. The latter two programs (primes-static.c and primes-dynamic.c) accept an additional command line argument - the chunk size (see starter code). What is the best chunk size for each type of scheduling? You must time each one of these implementations (including the sequential version). Talk to your TA about your findings and discuss the potential source of the differences you noticed.


    Timing the code: You must use the builtin profiling function omp_get_wtime() to measure execution time. See the OpenMP documentation for details on how to use this function. Execution times must be reported in milliseconds.

    3. Running the code:

    For running Lab04 on Scinet compute nodes two scripts called  run-job-lab04-prime.sh and run-job-lab04-integral.sh are provided in order to run the sequential and parallel implementation for computing an integral of a function and counting number of primes. You can call any of these scripts by putting ./ behind it.

    4. Submission

    Ensure that you have a repository directory created by MarkUs for this exercise. In this directory, add and commit+push all the files required to compile and run your code (all the source code and the makefile). You shouldn't need to add any new files to the starter code, but if you do, make sure that you update the makefile accordingly. Make sure your code compiles and runs correctly on the Scinet cluster.
